# 个人报销系统长期记忆

最后更新：2026-07-16

## 项目概况

- 核心目标：将多平台账单统一解析，筛选公司消费，生成报销表并可同步飞书多维表格。
- 当前阶段：本地 React MVP 已可运行。
- 主要用户：Musk。
- 代码仓库：本地 Git 仓库，当前无远程地址；已有交接设计文档提交，但大部分源码与用户资产仍未跟踪。
- 运行环境：本地 Vite 前端和本地飞书 API。

## 已确认的业务规则

- 支持 `.xlsx`、`.xls` 和 `.csv` 账单。
- 核心统一字段包括消费时间、金额、商户或摘要、支付账户和来源平台。
- 公司消费由用户确认，并可补充报销项目、费用类别和备注。
- 自动规则只用于辅助筛入和补全空白分类，不能覆盖用户已经填写的项目、费用类别或最终选择。
- 当前筛入规则覆盖大额消费、携程订单、工作日餐饮和高速费；规则只作用于当前筛选结果。
- 报销结果可按“节假日前后高速”规则筛除，并应保留最近一次对应操作的复原能力。
- 报销结果可导出 Excel/CSV，并支持同步飞书多维表格。
- 凭证图片当前可以使用结构化记录渲染，不要求第一阶段截取原 Excel 行。

## 架构与工具决策

### 2026-06-20 本地优先的 MVP

- 决策：使用 React、Vite 和本地 Node/TypeScript 服务。
- 原因：优先保证本地可运行和快速迭代，不引入不必要的云端架构。
- 对用户的影响：安装依赖后即可本地使用；飞书同步需要本地配置凭据。

### 2026-06-20 字段映射集中维护

- 决策：不同平台字段兼容集中在 `src/config/fieldMappings.ts`。
- 原因：避免平台判断分散到页面和导出逻辑。
- 对用户的影响：新增账单平台时更容易维护，旧平台行为更稳定。

### 2026-06-20 分层保存项目上下文

- 决策：`MEMORY.md` 只保存长期决策，当前功能和交互现状统一记录在 `PROJECT_CONTEXT.md`。
- 原因：避免长期记忆被可从代码读取的细节和聊天过程淹没，同时让新对话能够快速接手。
- 对用户的影响：后续换对话或项目路径时，不再依赖单条超长聊天恢复上下文。

### 2026-06-20 报销进度采用多入口恢复

- 决策：同时保留手动上传旧报销结果、固定文件一键恢复和浏览器本地进度恢复。
- 原因：固定文件适合快速继续上一期结果，手动上传提供兜底，本地草稿保存当前筛选和编辑状态。
- 对用户的影响：关闭页面或切换会话后，可以少做重复筛选和填写；浏览器本地进度不等同于跨设备持久化。

### 2026-06-20 飞书同步保持幂等

- 决策：通过稳定的 `同步ID` 判断更新或新增，并分别返回新增、更新和失败数量。
- 原因：重复同步不能生成重复报销记录，部分失败也必须对用户透明。
- 对用户的影响：同一批记录可以安全重复同步；权限或字段问题会显示明确失败结果。

### 2026-07-10 macOS 源码交接采用隐私白名单

- 决策：跨电脑交接使用源码、锁文件、项目上下文、模型部署入口和朋友使用说明组成的 macOS 本机包；打包采用白名单并自动检查禁止内容。
- 原因：另一个模型需要完整上下文才能可靠部署，但真实账单、历史结果、验证截图、Git 历史和飞书凭据不能随包迁移。
- 对用户的影响：同一个压缩包可以同时交给部署模型和朋友；新电脑需要重新安装依赖，且不会继承浏览器本地进度、历史结果或飞书配置。

## 已知问题与风险

- 大部分源码与现有用户资产仍未进入安全基线提交，版本历史保护尚不完整。
- 项目包含真实账单、报销结果和大量验证截图；处理时必须区分测试资料与个人隐私数据。
- `.env.local` 可能包含飞书凭据，不能提交或复制到文档。
- 当前只有交接包规则自动检查，业务功能仍主要依赖 lint、build 和手工流程验证。
- 真实账单（支付宝/微信/银行卡）导出格式与 `sample-data/` 模拟样本差异极大：CSV 为 CRLF 行尾、表头前带导出人/账户/特别提示等注释行、多为 GBK/GB18030 编码、表头字段名也不同（如支付宝真实为 `交易时间`/`商品说明`/`收/付款方式`/`金额`）。`src/utils/parseBills.ts` 的 `decodeCsv` 已统一去 `\r` 修复「整表被合并成单列导致原始 0 行」的 bug；字段映射集中在 `src/config/fieldMappings.ts`，已覆盖主流真实表头。新增平台兼容优先在此文件维护。金额为 `0.00` 的冲正/记账记录会被自动跳过（不导入），属预期行为。
