## 产品目标与工作流

这是一个本地运行的个人报销自动化工具，处理微信、支付宝、银行卡和信用卡账单。当前界面分为三个主要模块：

1. 上传账单
   - 支持同时上传 `.xlsx`、`.xls`、`.csv`。
   - 自动识别来源平台并统一字段。
   - 可上传以前导出的报销结果，恢复勾选和编辑内容。
   - 可一键读取 `public/reimbursement-results/个人报销表-2026-06-02.xlsx`。
   - 可从浏览器本地存储恢复用户主动保存的工作进度。
2. 消费筛选
   - 展示统一后的消费记录，支持表头筛选、包含/反选、金额排序和列显示调整。
   - 用户最终确认哪些记录属于公司消费。
   - 公司消费可编辑报销月份、报销人、项目、费用类别和报销备注。
   - 可按规则对当前筛选结果批量筛入公司消费。
   - 等级二判定洞察：每条记录展示「命中理由」与高/中/低「置信度」；「一键筛入」旁实时显示「影响预览（将新增 N 条 · ¥X）」；工具栏「边界复核」入口挑出最可能误判的记录请人工确认；根据手动勾选习惯弹「发现候选规则」可一键存为规则。
3. 报销结果
   - 展示已选公司消费并允许继续编辑。
   - 支持结果筛选、按规则批量筛除以及撤销最近一次对应规则的筛除。
   - 支持导出 Excel、CSV，生成结构化凭证图片，并同步飞书多维表格。

## 统一字段与解析约定

- 核心账单字段包括：消费时间、金额、交易类型、交易对方、商品说明、账单备注、支付账户和来源平台。
- 平台字段兼容集中维护在 `src/config/fieldMappings.ts`，不得把新的平台判断散落到页面、导出或同步逻辑。
- 每条记录保留来源文件、原始行号和原始字段，用于追溯和生成凭证。
- 解析和文档处理时只使用完成任务所需的信息，不展示或复制真实账单中的无关隐私。
- 真实账单解析注意事项（2026-07-15 修复验证）：支付宝/微信/银行卡导出的真实 CSV 与 `sample-data/` 模拟样本格式差异大——CSV 为 CRLF 行尾、表头前带导出说明/特别提示等注释行、多为 GBK/GB18030 编码、表头字段名也不同。`src/utils/parseBills.ts` 的 `decodeCsv` 已统一去除 `\r` 以修复「分隔符检测失败导致整表合并成单列、原始 0 行」的问题；字段映射集中在 `src/config/fieldMappings.ts` 并已覆盖主流真实表头。金额为 `0.00` 的冲正/记账记录会被自动跳过，属预期行为。

## 自动分类与筛入规则

系统会根据交易类型、交易对方、商品说明、账单备注和商户文本推荐报销项目与费用类别。已有手工填写值优先，自动分类只补空值。

当前可对“消费筛选”的当前筛选结果应用以下规则：

- 大额消费：金额大于 `1000` 元。
- 携程订单：文本包含携程、程支付、赫程或华程。
- 工作日餐饮：分类为餐饮、发生在周一至周五且金额大于 `100` 元。
- 高速费：文本包含高速、高速公路、通行费、过路费或 ETC，且金额不低于 `10` 元。

自动筛入只是辅助选择，用户仍可取消勾选并修改分类。

## 判定洞察（等级二）

等级二让报销判定「筛得更准、更可控」，全部由 `src/utils/selectionInsight.ts` 的纯函数派生，不写入本地进度草稿（旧备份结构不变、可正常恢复）。

- 命中理由：消费筛选「公司」列与报销结果「摘要」列展示每条记录的判定依据（命中了哪条筛入规则 / 人工勾选 / 规则命中却未勾选）。
- 置信度：每条记录给出高/中/低三档圆点。基准分按规则明确度设定（携程/大额最高，工作日餐饮/转账最低）；无规则命中的人工勾选视为用户确认（高）。「人工勾选无规则」「规则命中却未勾选」会被显式标记。
- 规则影响面预览：「一键筛入」按钮实时显示「将新增 N 条 · ¥X」，应用前先看波及范围，避免误伤。
- 边界主动复核：工具栏「边界复核 (N)」入口，按风险排序挑出最可能误判的 N 条（可能漏选 / 可能多选 / 低置信度），每条可「采纳」或「忽略」。
- 从行为学规则：分析手动勾选 / 未勾选的模式，对高支持度且高一致性的关键词归纳出候选规则，弹「发现候选规则」横幅，可一键「存为规则」并入自定义规则。

判定为辅助信息，最终公司消费选择与报销字段仍由用户控制。

## 报销结果筛除规则

当前规则为“节假日前后高速”：

- 作用范围只限当前报销结果。
- 匹配法定节假日和周末，以及其前后各两天内的高速差旅费。
- 每次规则筛除都会保存本轮记录 ID，可复原最近一次对应规则的筛除。

节假日、调休和星期显示依赖 `src/config/holidayWindows.ts`，修改日期规则时需同时验证“休”“班”标记与筛除结果。

## 恢复与持久化

系统有三种不同用途的恢复入口：

- 上传上次报销结果：用户手动选择以前导出的 Excel、CSV 文件。
- 恢复上一次报销结果：读取项目内固定的报销结果文件，并直接进入报销结果页。
- 恢复本地进度：读取浏览器 `localStorage` 中键为 `personal-reimbursement-progress-v1` 的草稿。

本地草稿包含消费记录、勾选状态、编辑字段、当前页面、筛选条件、排序、列顺序、列显示状态和规则操作历史。它只存在于当前浏览器，不是跨设备数据库。

## 表格与交互约定

- 表格按 Excel 式交互设计：筛选按钮靠近列名，筛选支持搜索、多选、包含和反选。
- 多列筛选采用级联选项：当前列的候选项受其他已启用筛选条件影响。
- 点击弹层外部应关闭筛选菜单；需要用户确认的筛选操作保留明确的确认入口。
- 列顺序可拖动，消费筛选和报销结果分别维护自己的列顺序和隐藏列。
- 日期和星期是独立列；日期输入框只容纳日期本身，不覆盖星期。
- 金额列保持紧凑，以实际金额长度为基准，不使用通用文本框宽度。
- 商品说明等长文本优先在单元格内可读，避免为了紧凑而永久截断关键信息。

## 导出、凭证与飞书同步

- Excel、CSV 导出需保持金额为数值、日期可读、中文字段完整。
- 凭证图片目前根据结构化记录渲染，不要求截取原始 Excel 行。
- 飞书同步由本地 `server/feishuServer.ts` 提供 API，凭据只从项目根目录 `.env.local` 读取。
- 服务会检查并补齐目标多维表字段。
- 每条记录使用稳定的 `同步ID`：飞书已有相同 ID 时更新，没有时新增。
- 同步结果必须分别报告新增、更新和失败数量；部分失败不能伪装成全部成功。
- 没有可用凭据或目标表权限时，只能报告未实际验证，不能声称同步成功。
